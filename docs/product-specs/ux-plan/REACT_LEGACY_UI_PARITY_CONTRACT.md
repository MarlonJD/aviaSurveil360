# React Legacy UI Parity Contract

## Status

Status: `candidate-only`, `release pending`.

This contract freezes the current React remediation scope for AviaSurveil360. It
does not claim stakeholder acceptance, production readiness, deployment,
traffic cutover, or legacy demo removal.

## User Outcome

The accepted root Vanilla interface remains the visual and behavioral oracle.
The current React candidate must make the 17 already-routed surfaces
recognizably match the accepted root workbench while preserving the React/Go
candidate authority, privacy, offline, and artifact boundaries.

This is not a full 86-screen React replacement. The committed source inventory
contains all 86 audited legacy rows, but only 17 rows are classified as
`react-parity` in this plan. The remaining 69 rows stay reachable only through
the intact root demo until Product approves a complete route-family slice.

## Binding Scope

- 86 ordered audit rows are extracted only from
  `docs/demo-evidence/UI_SCREEN_AUDIT_2026-07-19.md`.
- 17 rows map to typed React route contracts under `apps/web/src/app/route-contracts.ts`.
- 69 rows keep `reactPath: null`, `reactSurfaceId: null`, and no reference
  screenshot IDs.
- `New Inspection Planning Intake` maps to Department Manager Planning plus
  `ui-audit-047` through `ui-audit-051`; those wizard rows remain
  `later-legacy-only`.
- No React create/intake control or placeholder route may be added for the
  intake flow in this plan.

## Source Precedence

1. `AGENTS.md` and product/security/data authority under `docs/product-specs/`.
2. Verified root behavior tests and accepted browser-scenario evidence.
3. Root `index.html`, `css/styles.css`, and `js/` behavior.
4. The ordered 86-row audit and critical `qa/screenshots/` evidence.
5. Historical plans or screenshots.

Visual copying never overrides authority, privacy, lifecycle, immutable
history, truthful action state, or offline recovery.

## Visual And Action Rules

- Automated screenshots use tracked, hash-verified legacy baselines before
  visual migration.
- Shell diff ratio must be at most `0.03`.
- `content-adapted` regions may use at most `0.08` only when predeclared with
  semantic and geometry evidence.
- Masks must be explicit, leaf-scoped, deny broad shell/body selectors, and
  cover at most 5% of viewport pixels.
- Every visible control must work through Backend, a declared local/offline
  boundary, navigation, form behavior, or visible local state, or be disabled
  with an exact reason.
- Toast-only feedback does not satisfy a labeled action.

## Privacy And Lifecycle

- Potential Finding conversion remains Lead Inspector authority.
- CAP acceptance is not Finding closure.
- CAP revisions, Evidence versions, checklist-template versions, and report
  versions remain immutable.
- Auditee projections must be organization-scoped and structurally omit
  `Internal CAA Note`.
- `Comment to Auditee` and `Internal CAA Note` remain separate.
- Report issue does not close Findings.
- Offline field work must distinguish local, pending, acknowledged, conflict,
  and rejected states.

## Future Route Promotion Rule

A legacy-only row can be promoted only by a reviewed slice that includes Product
authority, OpenAPI, generated transport, mock and Go persistence/authorization,
React route/UI, mock and HTTP browser tests, privacy tests, visible-action
evidence, updated manifest disposition, and synchronized English/Turkish
evidence. Route-name parity alone is insufficient.
