# UX Principles

## 1. One screen, one main purpose

Every screen must answer one question. If a screen tries to plan audits, review CAPs and show analytics at the same time, split it.

## 2. Show next action first

Every audit/finding/CAP/evidence record should show:

- Current owner
- Next action
- Due date
- Days remaining or overdue
- Blocking reason
- Primary action button

## 3. Role-based navigation

Inspector menu: Dashboard, Audits, Findings, Organizations, Reports.

Auditee menu: My Findings, CAPs, Evidence, Messages, Reports.

Manager menu: Dashboard, Risks, Findings, Surveillance Plan, Reports.

Admin menu: Templates, Users, Settings, Audit Log.

## 4. Do not expose internal complexity

Avoid primary navigation labels like Workflow Engine, Taxonomy, Evidence Repository, Notification Engine and System Parameters for normal users.

## 5. CAP accepted is not closure

Use lifecycle stepper:

1. Finding issued
2. CAP submitted
3. CAP accepted
4. Evidence submitted
5. Evidence verified
6. Closed

## 6. Use plain language for auditee

Root cause: Why did this happen?

Corrective action: What will you do to fix it?

Preventive action: What will you change so it does not happen again?

Evidence: Upload files that show the action has been completed.

## 7. Dashboard must be actionable

Every dashboard card should link to a list or action. If a metric does not lead to action, put it in reports or remove it.
